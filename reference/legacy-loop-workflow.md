# 个人 Loop Engineering 工作流

## 1. 目标

把“卡墙上属于我的卡”和“Bug 中属于我的 Bug”转成一个可重复运行、可追踪、可验证的个人交付循环。

这个 loop 的目标不是让 Agent 自动替代判断，而是让 Agent 稳定承担发现、整理、澄清、拆解、执行辅助、验证和记录工作；最终交付质量仍由工程师确认。

## 2. 输入

### 2.1 卡墙上属于我的卡

来自项目卡墙或任务系统中分配给当前工程师的卡片。

需要识别的信息：

- 卡片标题、链接、状态、优先级、截止时间。
- 卡片类型：需求、优化、技术债、联调、验证、其他。
- 卡片业务背景和验收标准是否清楚。
- 是否已有设计、接口、测试说明或关联文档。
- 是否阻塞、依赖他人，或需要先向产品/后端/前端/测试确认。

### 2.2 Bug 中属于我的 Bug

来自缺陷系统、测试反馈、线上反馈或群消息中明确归属当前工程师的 Bug。

需要识别的信息：

- Bug 标题、链接、严重程度、优先级、复现环境。
- 复现步骤、实际结果、期望结果。
- 影响范围：页面、接口、数据、权限、兼容性、性能、线上风险。
- 是否可稳定复现。
- 是否已有日志、截图、录屏、接口响应或报错堆栈。

## 3. 状态文件

状态必须写在文件里，不能只留在对话上下文中。

推荐在云桌面代码仓库或对应功能目录下维护：

```text
.project/<feature-name>/00_requirements.md
.project/<feature-name>/01_design.md
.project/<feature-name>/02_plan.md
.project/<feature-name>/03_loop_state.md
```

如果某张卡或某个 Bug 很小，可以不强制创建完整三段文档，但仍应在 `03_loop_state.md` 中记录处理状态、验证结果和遗留问题。

## 4. Loop 总览

```mermaid
flowchart TD
    A["发现输入：我的-卡 / 我的 Bug"] -> B["分诊：是否可直接执行"]
    B --> C["需求澄清：00_requirements.md"]
    C --> D["技术设计：01_design.md"]
    D --> E["执行计划：02_plan.md"]
    E --> F["开发 / 修复"]
    F --> G["自动化测试"]
    G --> H["Chrome MCP 手动验证"]
    H --> I["记录状态：03_loop_state.md"]
    I --> J{"是否完成"}
    J -- "否" --> B
    J -- "是" --> K["交付 / 更新卡片或 Bug 状态"]
```

## 5. 阶段设计

### 5.1 Discover：发现工作

目标：定期收集“属于我的工作”，不要靠记忆捞任务。

输入：

- 卡墙上 assignee 是我的卡。
- Bug 系统中 assignee 是我的 Bug。
- 必要时补充群消息、测试反馈、PR 评论和 CI 失败。

输出：

- 今日候选工作清单。
- 新增、状态变化、阻塞变化和需要关注的高优先级项。

推荐记录到：

```text
.project/_inbox/YYYY-MM-DD.md
```

### 5.2 Triage：分诊与排序

目标：判断每个输入项应该进入哪个处理路径。

分诊结论：

- 直接执行：需求和技术边界足够清楚，直接进入 `02_plan.md` 或现有计划。
- 需求澄清：业务目标、用户故事或 AC 不清楚，进入 `story-requirement-clarification`。
- 技术设计：需求清楚但实现边界、接口、数据或权限不清楚，进入 `technical-design-clarification`。
- Bug 复现：缺少稳定复现路径，先补复现记录。
- 阻塞等待：依赖他人、环境、权限或数据，记录阻塞原因和下一步。
- 不处理：不属于我、重复、已关闭或需要转派。

排序建议：

1. 线上或高严重级 Bug。
2. 阻塞他人的卡或 Bug。
3. 当前迭代承诺交付的主流程卡。
4. 已经澄清和设计完成、可以快速闭环的任务。
5. 技术债、优化和低优先级补充项。

### 5.3 Clarify：需求澄清

触发条件：

- 卡片只有标题或口头描述。
- AC 不可测试。
- 输入 Story 需要拆分、合并、改名或重排。
- Bug 期望行为不清楚，可能其实是需求缺口。

使用 skill：

```text
story-requirement-clarification
```

输出：

```text
.project/<feature-name>/00_requirements.md
```

质量门：

- 每个纳入 Story 都有 `As a / I want / So that`。
- AC 使用 Given / When / Then。
- AC 能直接派生自动化测试或 Chrome MCP 手测用例。
- 不清楚的信息明确标为待确认。

### 5.4 Design：技术设计

触发条件：

- 需求和 AC 已确认。
- 需要明确前端、后端、API、数据、权限、异常、一致性、可观测性。
- Bug 修复可能影响已有接口、状态、数据或兼容性。

使用 skill：

```text
technical-design-clarification
```

输出：

```text
.project/<feature-name>/01_design.md
```

质量门：

- 每个 AC 都能映射到前端、后端、API、数据影响和测试关注点。
- 关键技术决策、假设、风险和被放弃方案有记录。
- 如果发现 AC 本身不成立，退回需求澄清，而不是在技术设计中偷偷补需求。

### 5.5 Plan：生成执行计划

触发条件：

- `00_requirements.md` 和 `01_design.md` 已确认。
- 准备进入开发或修复。

使用 skill：

```text
implementation-plan
```

输出：

```text
.project/<feature-name>/02_plan.md
```

质量门：

- 计划以执行原则开头。
- 任务可逐项勾选。
- 任务覆盖开发、自动化测试、集成/回归测试和 Chrome MCP 手动验证。
- 每个任务都有依据、完成标准和验证方式。

### 5.6 Execute：开发或修复

目标：按 `02_plan.md` 执行，而不是边写边扩大范围。

执行规则：

- 一次只推进一个明确任务。
- 完成一个任务后立即验证并标记。
- 没有验证证据，不勾选完成。
- 遇到需求或设计不成立，暂停并退回前置阶段。
- 对 Bug 修复，必须保留复现前后的对比证据。

### 5.7 Verify：验证

验证由三层组成：

- 自动化测试：单元测试、接口测试、集成测试或回归测试。
- 静态检查：lint、类型检查、构建检查或代码扫描。
- Chrome MCP 手动验证：按真实用户路径操作页面，观察页面状态、提示、网络响应和数据变化。

Chrome MCP 手测必须来自 AC 或 Bug 复现步骤，不写“看一下页面是否正常”这类不可执行任务。

### 5.8 Review：复盘与交付

完成前检查：

- 卡片或 Bug 的验收点是否全部覆盖。
- 是否有新增风险、兼容性影响或遗留问题。
- 是否需要更新卡墙状态、Bug 状态、评论或交付说明。
- 是否需要补充到 skill、项目约定或工作日志。

完成后记录到：

```text
.project/<feature-name>/03_loop_state.md
```

## 7. 人工确认点

以下节点不建议完全自动化跳过：

- 输入 Story 被拆分、合并、改名、重排或暂不纳入时。
- AC 影响业务范围或验收口径时。
- 技术设计引入新表、新接口、新权限、新外部依赖或兼容性风险时。
- Bug 修复方案存在多种取舍，且可能影响历史数据或其他流程时。
- Chrome MCP 手测失败但自动化测试通过时。
- 准备关闭卡片或 Bug 前。

## 8. 自动化建议

可以先从半自动 loop 开始：

1. 每天早上读取“我的卡”和“我的 Bug”，生成 inbox。
2. Agent 做分诊和排序，但不自动改状态。
3. 对需要澄清的项调用需求澄清 skill。
4. 对可执行的项生成或更新 `02_plan.md`。
5. 执行中由 Agent 辅助开发、测试和 Chrome MCP 手测。
6. 每天下班前汇总完成项、阻塞项和明日首要任务。

后续如果卡墙和 Bug 系统有 MCP 或 API，再把 Discover 和状态更新自动化。

## 9. 反模式

避免这些做法：

- 直接把卡片标题丢给 Agent 写代码。
- Bug 没有复现步骤就开始猜修复。
- AC 不清楚时用技术设计补业务口径。
- 计划里只有开发任务，没有测试和 Chrome MCP 手测。
- Agent 自己实现、自己判断完成、自己关闭卡片。
- 状态只存在对话里，第二天无法接上。
- 为了自动化而跳过工程师 review。

## 10. 最小可用版本

第一版 loop 可以很轻：

```text
我的卡 / 我的 Bug
-> 今日 inbox
-> 分诊排序
-> 需要澄清则写 00_requirements.md
-> 需要设计则写 01_design.md
-> 可执行则写 02_plan.md
-> 开发与验证
-> 记录 03_loop_state.md
-> 更新卡片或 Bug
```

先让状态文件和验证闭环跑顺，再逐步接入自动化、连接器和子 Agent。
